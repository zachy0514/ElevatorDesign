# Elevator System Simulation - Take-Home Solution

## Problem Understanding
This project simulates a **destination-dispatch** elevator system in discrete time.

Key requirements from the prompt:
- Serve all requests eventually (no indefinite waiting)
- Minimize passenger total time where
  - `total_time = wait_time + travel_time`
- Respect elevator constraints (capacity and direction behavior)
- Do not peek ahead at future requests
- Log elevator positions for each time step
- Produce passenger summary statistics (min/max/avg wait and total times)

## Approach
I implemented a tick-based simulation. At each time unit:
1. Add requests whose `time == current_time`
2. Assign each incoming request to a specific elevator (destination dispatch)
3. Process drops/pickups at current floors with capacity constraints
4. Move each elevator one floor toward its next target

### Scheduling Algorithm
The default scheduler is a **nearest-car style greedy heuristic** that scores each elevator by:
- Distance to pickup floor
- Light workload factor (pending pickup and dropoff stops)

This keeps the behavior easy to explain while still avoiding obvious overload on a single elevator.

Additional scheduler modes are included:
- `nearest` (default)
- `strict_nearest` (distance-only baseline)
- `round_robin` (fair rotational baseline)

## Project Structure
- `main.py`: CLI entrypoint
- `elevator_sim/models.py`: request/elevator state models
- `elevator_sim/scheduler.py`: scheduler logic
- `elevator_sim/simulation.py`: simulation engine + file outputs
- `sample_requests.csv`: sample input file
- `compare_schedulers.py`: runs all scheduler modes and compares metrics
- `plot_results.py`: creates charts from generated CSV outputs
- `tests/test_simulation.py`: basic correctness test

## How To Run
### 1) Run simulation
```bash
python main.py --input sample_requests.csv --num-elevators 3 --num-floors 60 --capacity 8 --scheduler nearest
```

Scheduler options:
- `--scheduler nearest`
- `--scheduler strict_nearest`
- `--scheduler round_robin`

### 2) Compare algorithms
```bash
python compare_schedulers.py --input sample_requests.csv --num-elevators 3 --num-floors 60 --capacity 8
```

### 3) Interactive visualization (optional)
```bash
python visualize.py
```
Opens `docs/index.html` — an animated presentation showing elevator movement,
passenger status cards, Gantt timeline, and summary stats.

**Live demo (GitHub Pages):** https://zachyang.github.io/ElevatorDesign

### 2) Run tests
```bash
python -m unittest discover -s tests -p "test_*.py"
```

## Outputs
After running, these files are generated by default:
- `output/elevator_positions.csv`
  - One row per time step: timestamp + all elevator floors
- `output/passenger_times.csv`
  - Per-passenger wait/travel/total time and assignment
- `output/summary.txt`
  - Min/max/avg stats and notable observations
- `output/scheduler_comparison.csv`
  - Side-by-side algorithm metrics for panel discussion
- `output/simulation_plots.png` (optional)
  - Position trajectory and time-distribution charts


## Assumptions, Simplifications, Trade-offs
- One floor traversal takes exactly one time unit.
- Boarding/alighting at a floor is modeled within that same time tick.
- Passenger destination is immutable after request (as required).
- Requests are assigned immediately on arrival and remain assigned to that elevator.
- The scheduler is greedy, not globally optimal.

Trade-off:
- Greedy assignment is easy to explain and fast, but can be suboptimal under heavy burst traffic compared to global optimization.
