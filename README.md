# Elevator System Simulation - Take-Home Solution

## Problem Understanding
This project simulates a **destination-dispatch** elevator system in discrete time.

Key requirements from the prompt:
- Serve all requests eventually (no indefinite waiting)
- Minimize passenger total time where
  - `total_time = wait_time + travel_time`
- Respect elevator constraints (capacity and direction behavior)
- Do not peek ahead at future requests
- Log elevator positions for each time step
- Produce passenger summary statistics (min/max/avg wait and total times)

## Approach
I implemented a tick-based simulation. At each time unit:
1. Add requests whose `time == current_time`
2. Assign each incoming request to a specific elevator (destination dispatch)
3. Process drops/pickups at current floors with capacity constraints
4. Move each elevator one floor toward its next target

### Scheduling Algorithm
The default scheduler is a **nearest-car style greedy heuristic** that scores each elevator by:
- Distance to pickup floor
- Light workload factor (pending pickup and dropoff stops)

This keeps the behavior easy to explain while still avoiding obvious overload on a single elevator.

Additional scheduler modes are included:
- `nearest` (default)
- `strict_nearest` (distance-only baseline)
- `round_robin` (fair rotational baseline)

## How To Run

### 1) Run simulation
```bash
python main.py --input sample_requests.csv --num-elevators 3 --num-floors 60 --capacity 8 --scheduler nearest
```

Scheduler options:
- `--scheduler nearest`
- `--scheduler strict_nearest`
- `--scheduler round_robin`

### 2) Compare algorithms
```bash
python compare_schedulers.py --input sample_requests.csv --num-elevators 3 --num-floors 60 --capacity 8
```

### 3) Interview presentation + visualization
```bash
python present.py
```
Generates `docs/index.html` — 7 content slides followed by a live animated visualization with building view, passenger cards, Gantt timeline, and summary stats.

**Live demo:** https://zachy0514.github.io/ElevatorDesign

### 4) Run tests
```bash
python -m unittest discover -s tests -p "test_*.py"
```

## Assumptions, Simplifications, Trade-offs
- One floor traversal takes exactly one time unit (no acceleration or door time).
- Boarding and alighting happen within the same tick the elevator arrives.
- Passenger destination is immutable after request (as required).
- Requests are assigned immediately on arrival and remain assigned to that elevator.
- The scheduler is greedy, not globally optimal.

**Trade-off — assignment score vs. actual arrival time:**

When a passenger requests an elevator, the scheduler picks the best one using a score: `distance to passenger + (pending stops × 2)`. Lowest score wins.

The problem is that distance is only a reliable proxy for arrival time when an elevator is idle. If it's already moving, the two can diverge significantly. For example: elevator A is at floor 4 heading up to floor 10, and a passenger requests floor 3. Score = 1 (very close) — but it won't arrive at floor 3 until it finishes going up and reverses, roughly 13 ticks away. An idle elevator B at floor 6 scores worse (distance 3) but arrives in 3 ticks.

The movement engine itself is direction-aware (it continues in the current direction before reversing). The gap is purely in the assignment step — it doesn't factor in which way an elevator is heading when deciding who to send.

## What I'd Improve With More Time

**Direction-aware assignment score:** Instead of raw distance, estimate actual ticks until the elevator reaches the passenger — accounting for current direction and remaining stops before it can reverse. All the data needed (`direction`, `pickup_floors`, `dropoff_floors`) is already on the elevator model; it just isn't used in the scorer.

**Capacity-aware assignment:** The scheduler does not check capacity when picking an elevator — it scores purely on distance and workload. A full elevator can be assigned a new passenger, travel to their floor, and fail to board them (capacity check only fires at pickup time). The fix is non-trivial: current load isn't reliable because passengers may drop off before the elevator arrives, so the real solution is to project future load at estimated arrival time.

**Realistic time model:** Currently boarding is instant — passengers appear and disappear in the same tick the elevator arrives. In reality there's door open/close time and acceleration, which meaningfully affects scheduling decisions.

**Zone-based routing:** In tall buildings, pin elevators to floor ranges (e.g. floors 1–20, 21–40) so they don't waste time traveling the full height for every request.

**Dynamic reassignment:** Once assigned, a passenger is locked to their elevator even if a much better one becomes free nearby. This would allow re-evaluating assignments each tick and switching if it'd save significant time.

**Stress testing:** The test suite only covers basic correctness on small inputs. There are no stress tests with high passenger volume, tight capacity, or edge cases like all passengers requesting the same floor simultaneously. These are exactly the conditions that expose scheduling bugs — such as the capacity deadlock that only triggers when an elevator is full at a pickup floor.
