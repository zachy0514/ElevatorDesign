The Problem

Three objectives from the spec, plus one hard constraint
  • Serve all requests eventually
    No passenger should wait indefinitely to be picked up or dropped off
  • Minimize total_time = wait_time + travel_time
    Destination is known upfront — destination dispatch — so the system can route optimally at assignment time
  • Honor elevator constraints: capacity and direction logic
    Elevators cannot exceed max passengers; movement follows SCAN — commit to current direction before reversing
  • No peeking at future requests
    Only admit a request when its arrival time equals the current tick — even if the input skips ahead in time

The Simulation Loop

One tick = one floor of travel = one time unit
1
Record positions
Snapshot all elevator floors at the start of the tick — written to the positions log
2
Assign new requests
Admit requests where time == current_tick — scheduler picks an elevator and adds the source floor to its pickup set
3
Process stops — dropoffs first, then pickups
Passengers who reached their destination alight; waiting passengers board up to capacity. Dropoffs before pickups so freed seats are available in the same tick
4
Move elevators
Each elevator steps one floor toward its next SCAN target. Terminate if all passengers served and no future requests remain
while not all_done:
  record() → assign() → process_stops() → move()

Schedulers

All three answer one question: which elevator do we send for this request? — Multiple schedulers were listed as a bonus item in the spec; all three are implemented and compared via compare_schedulers.py
  • nearest default
    score = distance_to_pickup + (pending_stops × 2) → lowest wins
    Penalizes busy elevators. Balances response speed with workload fairness.
  • strict_nearest
    min(elevators, key=lambda e: abs(e.floor - request.source))
    Distance only. Can overload the closest elevator while others sit idle.
  • round_robin
    elevators[counter % N]; counter += 1
    Maximum fairness. Ignores location entirely.

SCAN — Elevator Movement

Commit to the current direction before reversing — like a scanner head sweeping across a page
  • Going UP
    Serve the lowest floor greater than or equal to the current position
  • Going DOWN
    Serve the highest floor less than or equal to the current position
  • Idle
    Pick the absolutely nearest floor — no directional bias
if UP:   return min(targets ≥ current)
if DOWN: return max(targets ≤ current)
else:    return nearest(targets)

Why SCAN?

Without it, an elevator heading to floor 10 would immediately reverse for a request at floor 3 — abandoning passengers already waiting above.

SCAN finishes the sweep first. Everyone going the same direction is served before the elevator reverses.

Trade-offs & What I'd Improve

  • Score does not reflect actual arrival time
    Distance is only reliable when an elevator is idle. A moving elevator may score low but arrive much later. Fix: estimate ticks to arrival using direction and remaining stops — all data is on the Elevator model.
  • Scheduler ignores capacity
    A full elevator can be assigned a passenger, travel to their floor, and fail to board them — capacity is only enforced at pickup time. Fix: project future load at estimated arrival time before assigning.
  • Instant boarding
    Passengers board and alight in the same tick the elevator arrives. Fix: model door open/close time and deceleration cost.
  • No dynamic reassignment
    Once assigned, a passenger stays with their elevator even if a much better one becomes free. Fix: re-evaluate assignments each tick and switch if the time savings exceed a threshold.
  • No zone-based routing
    All elevators serve all floors — wasteful in tall buildings. Fix: pin elevators to floor ranges with a dedicated express elevator strategy.
